# LabelWe 工程实现决策

## 目标

本文件是对 [图像检测协同标注平台-软件描述.md](e:/work/project/labelwe/docs/图像检测协同标注平台-软件描述.md) 的工程化落地补充，记录本次仓库实现中采用的技术栈、边界和当前环境下的验证策略。

## 选型

- 后端：FastAPI + SQLAlchemy 2.x + SQLite 本地开发模式 / MySQL 8 Docker 部署模式
- 前端：React 18 + TypeScript + Vite
- 认证：自定义 HMAC Bearer Token，避免当前环境额外引入 JWT 库
- 密码：标准库 `pbkdf2_hmac`
- 导出：Python 标准库 `zipfile`
- 测试：Pytest + FastAPI TestClient；Playwright 作为浏览器级补充验证

## 为什么这样实现

- 当前仓库是从规格直接起步，FastAPI 能较快形成清晰的 OpenAPI 风格 API。
- SQLite 让当前非 Linux 环境也能快速做可执行验证；生产部署默认使用 MySQL 8，便于与容器化环境和桌面数据库工具联通。
- React + Vite 可以更快实现标注器、审核台和管理端的交互闭环。
- `prototype/` 先承载复杂交互原型，`frontend/` 再承载正式工程，以降低视觉和交互探索对主工程的扰动。

## 与规格的对齐方式

- 非任务拥有者写入工作副本时，服务端返回 `403 FORBIDDEN_NOT_ASSIGNEE`。
- 待审提交存在时，工作副本写入返回 `409 TASK_IN_REVIEW`。
- “无目标 / 空标注”通过 `annotation_state + is_no_object` 显式表示。
- 同一源图在未完成任务中排他分发；若已被占用，创建或追加任务图像返回 `409 SOURCE_IMAGE_ALREADY_ASSIGNED`。
- 审核通过与部分通过统一写入 `effective_annotations`，历史记录通过 `superseded_at` 保留。

## 当前环境中的已知限制

- 本环境不是目标 Linux 部署环境，因此 Docker 容器、Nginx 代理和生产权限模型只能做静态交付，不能视为最终上线验证。
- `web-artifacts-builder` 的 bash 脚本在当前环境中不可直接执行，因此原型采用“技能设计原则 + 仓库内独立 prototype 文件”的方式实现。
- Playwright 浏览器安装若受网络或权限限制，浏览器级验收将保留脚本并尽量执行；失败时会在最终说明中记录未完成项。
